# Argus 项目文档 V4.0

> Argus 是一个 **RAG 知识库平台** 的教学项目，从零搭建，逐步迭代。
> V4.0 在 V3.0（知识问答）的基础上，构建了**交互式 AI 助手模块**，引入 Agent 架构、短期记忆管理和流式对话，
> 使平台从"一次性问答"升级为**具备上下文感知能力的多轮对话系统**。

---

## 一、项目概述

### 1.1 V4.0 定位

V3.0 完成了"文档上传 → 自动解析索引 → 混合检索 → AI 生成回答"的完整闭环。但 V3.0 的问答是**无状态的**——每次提问独立处理，没有对话历史，没有上下文记忆。

V4.0 在此基础上构建了 **AI 助手（Assistant）模块**，核心能力包括：

- **多轮对话**：用户与助手可以持续对话，助手记住上下文。
- **Agent 架构**：基于 Spring AI Alibaba ReactAgent 图执行引擎，支持"思考 → 工具调用 → 生成回复"的完整链路。
- **双模式切换**：纯对话模式（CHAT）和知识库检索模式（KB_SEARCH），同一会话内可动态切换。
- **短期记忆管理**：三级压缩策略（会话记忆 → 紧凑摘要 → 运行时压缩），在有限上下文窗口内维持长对话。
- **SSE 流式输出**：模型回复逐字推送给前端，用户无需等待完整生成。

V4.0 的核心创新集中在**对话管理侧**：用 ReactAgent 统一驱动两种模式，用 LLM 驱动的记忆压缩替代简单的滑动窗口，用 BEFORE_MODEL Hook 实现上下文的无侵入注入。

### 2.2 V4.0 新增依赖

V4.0 引入了 **一项新的 Maven 依赖**：

```xml
<dependency>
    <groupId>com.alibaba.cloud.ai</groupId>
    <artifactId>spring-ai-alibaba-agent-framework</artifactId>
    <version>1.1.2.0</version>
</dependency>
```

该依赖提供了 `ReactAgent`（图执行引擎）、`MessagesModelHook`（模型调用前后拦截）、`RunnableConfig`（图节点间元数据传递）、`CompileConfig`（递归上限控制）和 `MemorySaver`（检查点保存）等核心基础设施。

V4.0 的其余能力——会话管理、消息持久化、上下文加载、短期记忆维护——均通过组合现有的 Spring Boot + MyBatis-Plus + Spring AI 基础设施实现，**未引入额外的第三方依赖**。

### 2.3 架构分层

```
Spring AI Alibaba Agent Framework (ReactAgent)
    ├── AssistantAgentFacade          ← Agent 调用门面（同步 + 流式）
    ├── AssistantReactAgentFactory    ← Agent 实例工厂（装配工具/Hook/配置）
    ├── AssistantShortTermMemoryHook  ← BEFORE_MODEL Hook（上下文注入）
    └── AssistantKnowledgeBaseTool    ← Spring AI @Tool（知识库检索）

Spring AI ChatClient
    ├── AssistantMemorySummarizer     ← 三级 LLM 摘要（session memory / compact / runtime）
    └── (DashScope ChatModel)         ← 底层大模型调用

Spring Boot + MyBatis-Plus
    ├── AssistantService              ← 聊天流程编排（同步 + 流式）
    ├── AssistantConversationService  ← 消息持久化 + 上下文加载
    ├── AssistantSessionService       ← 会话 CRUD
    └── AssistantShortTermMemoryMaintenanceService ← 记忆阈值判断 + 触发维护

V3.0 检索基础设施（复用）
    ├── QaRetrievalService            ← 检索门面（统一对外接口）
    ├── ReadyChunkDocumentRetriever   ← 混合检索适配器（QaRetrievalService 内部委托）
    ├── HybridChunkRetrievalService   ← RRF 混合检索引擎
    └── CitationAssembler             ← 引用来源组装器（QaRetrievalService 内部调用）
```

### 3.2 模块层次说明

Assistant 模块遵循与 V3.0 相同的分层架构，但新增了 Agent 层：

```
controller (AssistantChatController / AssistantSessionController / AssistantConversationController)
    ↓ 校验输入 + 提取用户身份
service (AssistantService)
    ↓ 参数校验 + 权限校验 + 编排
agent (AssistantAgentFacade)
    ↓ 构建 instruction + RunnableConfig → 调用 ReactAgent
memory (AssistantShortTermMemoryHook)
    ↓ BEFORE_MODEL 阶段注入上下文
    ↓ 加载 compact summary → session memory → recent messages
service (AssistantConversationService)
    ↓ 消息持久化 + 上下文加载 + 摘要触发
memory (AssistantShortTermMemoryMaintenanceService)
    ↓ 判断阈值 → 触发 LLM 摘要 → 乐观锁写回
```

## 四、数据库设计

### 4.1 概述

**V4.0 新增了 3 张数据库表**，用于支撑多轮对话的会话管理、消息持久化和上下文记忆。

| 表名 | 用途 | 核心字段 |
|------|------|---------|
| `assistant_sessions` | 会话管理 | id, user_id, title, status, last_message_at |
| `assistant_messages` | 消息持久化 | id, session_id, role, tool_mode, group_id, content, structured_payload |
| `assistant_session_contexts` | 短期记忆存储 | session_id, session_memory, compact_summary, summary_text, context_version |

### 4.2 assistant_sessions（会话表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGSERIAL | PK | 会话主键，自增 |
| user_id | BIGINT | NOT NULL | 所属用户 ID，关联 `users.id` |
| title | VARCHAR(255) | NOT NULL | 会话标题，默认"新会话" |
| status | VARCHAR(32) | NOT NULL | 会话状态：ACTIVE / ARCHIVED / DELETED |
| last_message_at | TIMESTAMP | NULL | 最后消息时间，用于排序展示 |
| created_at | TIMESTAMP | NOT NULL | 创建时间 |
| updated_at | TIMESTAMP | NOT NULL | 更新时间 |

设计要点：
- `status` 使用软删除模式（`DELETED` 状态），所有查询均过滤 `status <> 'DELETED'`，删除操作执行物理删除。
- `last_message_at` 允许 NULL，新创建的会话尚无消息时为空，查询时 `ORDER BY last_message_at DESC NULLS LAST`。
- 通过 `(id, user_id)` 联合校验确保用户只能操作自己的会话。

### 4.3 assistant_messages（消息表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGSERIAL | PK | 消息主键，自增 |
| session_id | BIGINT | NOT NULL | 所属会话 ID，关联 `assistant_sessions.id` |
| role | VARCHAR(16) | NOT NULL | 消息角色：USER / ASSISTANT / TOOL |
| tool_mode | VARCHAR(32) | NOT NULL | 消息产生时的工具模式：CHAT / KB_SEARCH |
| group_id | BIGINT | NULL | 知识库组 ID，CHAT 模式下为 NULL |
| content | TEXT | NOT NULL | 消息文本内容 |
| structured_payload | JSONB | NULL | 结构化负载（如工具调用结果 JSON） |
| created_at | TIMESTAMP | NOT NULL | 消息创建时间 |

设计要点：
- `structured_payload` 使用 PostgreSQL `jsonb` 类型，MyBatis 插入时通过 `cast(#{structuredPayload} as jsonb)` 显式转换。
- `group_id` 仅在 `tool_mode = 'KB_SEARCH'` 时有值，CHAT 模式下为 NULL。
- 消息按 `created_at` 升序排列用于上下文加载，按 `created_at DESC` + `LIMIT` 取最近 N 条。
- 索引建议：`(session_id, created_at)` 复合索引覆盖最频繁的查询模式。

### 4.4 assistant_session_contexts（会话上下文表）

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| session_id | BIGINT | PK | 会话 ID，一对一关联 `assistant_sessions.id` |
| session_memory | TEXT | NULL | 会话记忆内容（LLM 生成的增量摘要） |
| compact_summary | TEXT | NULL | 紧凑摘要（更精炼的会话压缩） |
| session_memory_base_message_id | BIGINT | NULL | 会话记忆覆盖的起始消息 ID |
| session_memory_range_end_message_id | BIGINT | NULL | 会话记忆覆盖的结束消息 ID |
| compact_summary_base_message_id | BIGINT | NULL | 紧凑摘要覆盖的起始消息 ID |
| compact_summary_range_end_message_id | BIGINT | NULL | 紧凑摘要覆盖的结束消息 ID |
| summary_text | TEXT | NULL | 会话摘要文本（早期消息的简明摘要） |
| source_message_id | BIGINT | NULL | 摘要覆盖的结束消息 ID |
| context_version | BIGINT | NOT NULL | 乐观锁版本号，初始为 0，每次更新递增 |
| updated_at | TIMESTAMP | NOT NULL | 上下文最后更新时间 |

设计要点：
- 使用 `INSERT ... ON CONFLICT (session_id) DO UPDATE`（PostgreSQL upsert）语义，首次写入和后续更新共用一条 SQL。
- `context_version` 实现乐观锁——更新时 WHERE 条件包含 `context_version = #{expectedVersion}`，更新失败表示并发冲突。
- `coalesce(excluded.xxx, assistant_session_contexts.xxx)` 确保只覆盖非 NULL 字段——允许部分字段更新（例如仅更新 `session_memory` 而不影响 `compact_summary`）。
- 多个 `_range_end_message_id` 字段用于追踪摘要覆盖的消息范围——下一次维护时，只处理 `messageId > lastRangeEndMessageId` 的新消息。

### 4.5 索引建议

| 索引 | 所在表 | 用途 |
|------|--------|------|
| `(user_id, status)` 复合索引 | `assistant_sessions` | 查询用户活跃会话列表 |
| `(session_id, created_at)` 复合索引 | `assistant_messages` | 按会话加载消息（增量/全量） |
| `session_id` 唯一索引（主键已覆盖） | `assistant_session_contexts` | 按会话查上下文 |

## 七、关键配置说明

### 7.1 短期记忆阈值配置

V4.0 的短期记忆维护依赖以下 `application.yml` 配置项（前缀 `assistant.short-term-memory`）：

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `assistant.short-term-memory.session-token-threshold` | 6500 | 触发会话压缩的总体 token 阈值 |
| `assistant.short-term-memory.session-memory-message-trigger` | 4 | 触发会话记忆更新的消息数阈值 |
| `assistant.short-term-memory.session-memory-token-trigger` | 1200 | 触发会话记忆更新的 token 数阈值 |
| `assistant.short-term-memory.compact-message-trigger` | 6 | 触发紧凑摘要的消息数阈值 |
| `assistant.short-term-memory.compact-token-trigger` | 1800 | 触发紧凑摘要的 token 数阈值 |

配置项通过 `@Value` 注入 `AssistantShortTermMemoryMaintenanceService` 构造函数：

```java
public AssistantShortTermMemoryMaintenanceService(
        ...
        @Value("${assistant.short-term-memory.session-token-threshold:6500}") int sessionTokenThreshold,
        @Value("${assistant.short-term-memory.session-memory-message-trigger:4}") int sessionMemoryMessageTrigger,
        ...
) { ... }
```

### 7.2 会话摘要配置

| 配置项 | 默认值 | 说明 |
|--------|--------|------|
| `assistant.session.summary.message-threshold` | 20 | 触发会话摘要的消息数阈值 |
| `assistant.session.summary.token-threshold` | 8000 | 触发会话摘要的 token 数阈值 |
| `assistant.session.summary.stale-days` | 7 | 摘要过期天数，超过此天数需重新生成 |

### 7.3 Agent 配置

Agent 的核心参数以常量形式定义在代码中：

| 参数 | 值 | 定义位置 | 说明 |
|------|----|---------|------|
| `AGENT_RECURSION_LIMIT` | 10 | `AssistantReactAgentFactory` | ReactAgent 图递归上限——KB_SEARCH 模式至少需要"决策→调工具→生成回答"三步，10 次递归足够应对复杂场景 |
| `RECENT_MESSAGE_LIMIT` | 10 | `AssistantShortTermMemoryHook` | 注入上下文的最近消息条数 |
| `RUNTIME_TOKEN_THRESHOLD` | 50000 | `AssistantShortTermMemoryHook` | 触发运行时压缩的 token 阈值 |

### 7.4 SSE 配置

| 参数 | 值 | 定义位置 | 说明 |
|------|----|---------|------|
| `SSE_TIMEOUT_MILLIS` | 0 | `AssistantChatController` | SSE 超时时间，0 表示无超时 |
| 线程池 | `CachedThreadPool` | `AssistantChatController` | SSE 推送线程池，无界缓存线程池 |

### 7.5 Token 估算

Token 估算采用**字符数 / 4** 的粗略方法（中文约 1.5 字符/token，英文约 4 字符/token，取 4 作为保守估计）：

```java
// AssistantShortTermMemoryMaintenanceService.java
private static final int TOKEN_ESTIMATE_DIVISOR = 4;

public int estimateTokens(List<AssistantMessageEntity> messages) {
    int totalChars = messages.stream()
            .map(AssistantMessageEntity::getContent)
            .filter(content -> content != null && !content.isBlank())
            .mapToInt(String::length)
            .sum();
    return Math.max(1, totalChars / TOKEN_ESTIMATE_DIVISOR);
}
```

## 八、枚举扩展

### 8.1 V4.0 新增枚举

| 枚举 | 值 | 用途 |
|------|----|------|
| `AssistantToolMode` | `CHAT`, `KB_SEARCH` | 对话工作模式，影响 Agent 的 system prompt 和可用工具 |
| `AssistantMessageRole` | `USER`, `ASSISTANT`, `TOOL` | 消息发送者角色，用于消息展示和历史加载 |
| `AssistantSessionStatus` | `ACTIVE`, `ARCHIVED`, `DELETED` | 会话生命周期状态 |

### 8.2 工具模式的行为差异

```
用户选择模式
    │
    ├── CHAT（纯对话）
    │   ├── Agent 不装配任何工具
    │   ├── system prompt 不追加知识库检索规则
    │   ├── groupId 必须为 null（请求校验层拒绝）
    │   └── Agent 直接基于模型知识 + 对话历史回答
    │
    └── KB_SEARCH（知识库检索）
        ├── Agent 装配 knowledgeBaseSearch 工具
        ├── system prompt 追加 6 条知识库检索规则
        ├── groupId 必须非 null（请求校验层拒绝）
        ├── Agent 自主决定是否调用工具
        ├── 每轮对话最多调用一次检索工具
        └── 工具返回证据后 Agent 基于证据生成回答
```

### 8.3 消息角色的使用场景

| 角色 | 产生时机 | 存储内容 |
|------|---------|---------|
| `USER` | 用户每次发送消息 | 用户输入的原始文本 |
| `ASSISTANT` | Agent 每次完成回复 | Agent 生成的最终回答文本 |
| `TOOL` | Agent 调用工具后 | 工具返回的结构化结果（JSON） |

`TOOL` 角色消息当前预留，实际工具调用结果通过 `BEFORE_MODEL Hook` 的 `appendRuntimeToolMessages` 注入上下文，不单独持久化为 TOOL 消息。

### 8.4 会话状态的生命周期

```
ACTIVE ──────→ ARCHIVED ──────→ DELETED
  │                                  │
  └──────────────────────────────────┘  (直接删除)
```

- **ACTIVE**：正常状态，可收发消息。
- **ARCHIVED**：用户归档，不在列表中展示但仍可查看。
- **DELETED**：逻辑删除状态（查询过滤），物理删除通过 DELETE 接口执行。
