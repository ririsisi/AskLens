# AskLens 启动流程与配置加载说明

> 本文档从零梳理 AskLens 项目启动时**加载了哪些配置文件、执行了哪些初始化流程、以及每一步为什么这么做**。

---

## 一、启动入口

```java
// AskLensBackendApplication.java
@SpringBootApplication
@EnableAsync
@MapperScan("com.asklens.**.mapper")
public class AskLensBackendApplication {
    public static void main(String[] args) {
        SpringApplication.run(AskLensBackendApplication.class, args);
    }
}
```

`SpringApplication.run()` 触发的是一条精心编排的启动链：

```
main(String[] args)
    → SpringApplication.run(AskLensBackendApplication.class, args)
        → prepareEnvironment()          ← 加载配置文件（阶段一）
        → prepareContext()              ← 注册 Bean 定义（阶段二）
        → refreshContext()              ← 实例化 Bean + 执行回调（阶段三）
        → callRunners()                 ← 执行 ApplicationRunner（阶段四）
        → 发布 ApplicationReadyEvent   ← 应用就绪
```

## 二、配置文件加载（阶段一：prepareEnvironment）

### 2.1 加载路径与优先级

Spring Boot 按照固定优先级依次加载以下位置的 `application.*` 文件（高优先级覆盖低优先级）：

```
优先级（低 → 高）：
1. classpath:application.yml                      ← 基础配置
2. classpath:application-{profile}.yml            ← profile 专用配置（本例：application-local.yml）
3. 外部化配置（环境变量、命令行参数等）              ← 未使用，但机制存在
```

### 2.2 application.yml — 基础配置层

```yaml
# application.yml — 所有 profile 共用的基础配置
spring:
  profiles:
    active: local          # ← 激活 local profile
mybatis-plus:
  mapper-locations: classpath*:/mappers/**/*.xml
  type-aliases-package: com.asklens
  configuration:
    map-underscore-to-camel-case: true
    default-enum-type-handler: org.apache.ibatis.type.EnumTypeHandler
```


| 配置项                                              | 为什么这么设                                              |
| ------------------------------------------------ | --------------------------------------------------- |
| `profiles.active: local`                         | 默认本地开发环境。团队其他成员通过 `--spring.profiles.active=dev` 覆盖 |
| `mapper-locations: classpath*:/mappers/**/*.xml` | `classpath*` 扫描所有 jar 包中的 mapper XML，支持未来模块化拆分      |
| `type-aliases-package: com.asklens`            | 全局别名，XML 中写 `UserEntity` 而非全限定类名                    |
| `default-enum-type-handler: EnumTypeHandler`     | 枚举以 `name()` 字符串存入数据库，可读且不随 ordinal 变化              |


**为什么分离 base 和 profile**：`application.yml` 存放 MyBatis-Plus、Jackson 等框架级通用配置（所有环境一致），`application-{profile}.yml` 存放数据库连接、API Key 等环境相关配置。

### 2.3 application-local.yml — Profile 配置层

`application-local.yml` 在 `application.yml` 之后加载，同名字段覆盖前者。其内容按配置前缀分组：

#### Spring 核心

```yaml
server:
  port: 10001
spring:
  application:
    name: asklens-service
```


| 配置项                       | 值             | 为什么                                                |
| ------------------------- | ------------- | -------------------------------------------------- |
| `server.port`             | 10001         | 避开 8080/9090 等常见端口，减少冲突                            |
| `spring.application.name` | asklens-service | 日志/监控中标识服务；Spring AI ChatClient 自动以此为 `user-agent` |


#### 数据源

```yaml
spring:
  datasource:
    url: jdbc:postgresql://192.168.200.130:5433/dd_rag
    username: postgres
    password: ...
    driver-class-name: org.postgresql.Driver
    hikari:
      connection-timeout: 5000
      maximum-pool-size: 10
      minimum-idle: 2
      idle-timeout: 300000
```


| 配置项                        | 为什么                                                                                     |
| -------------------------- | --------------------------------------------------------------------------------------- |
| `maximum-pool-size: 10`    | 当前为开发环境单机使用，10 足够。生产环境按 `connections = ((core_count * 2) + effective_spindle_count)` 调大 |
| `minimum-idle: 2`          | 保持少量热连接，避免冷启动等待；2 已覆盖低频轮询场景                                                             |
| `connection-timeout: 5000` | 5 秒超时快速失败，避免线程卡死在等待连接上                                                                  |


#### LLM 配置（Spring AI）

| 决策                                | 为什么                                                                            |
| --------------------------------- | ------------------------------------------------------------------------------ |
| `access-token-expire-minutes: 30` | 访问令牌短周期，泄漏后影响窗口小                                                               |
| `refresh-token-expire-days: 14`   | 刷新令牌长周期，避免频繁重新登录                                                               |
| `refresh-cookie-secure: false`    | local 环境无 HTTPS，设为 false 允许明文传输。生产环境必须 true                                    |

#### Elasticsearch

```yaml
elasticsearch:
  uris: http://192.168.200.130:9200
  chunk-index: dd_rag_document_chunks
```

Elasticsearch 不使用 Spring Data Elasticsearch 而用 JDK `HttpClient` 直连。原因：Spring Data ES 的 `ElasticsearchOperations` 抽象重且版本绑定紧密，而项目只需要 `_search` + `_bulk` + `_delete_by_query` 三个操作，HTTP 直连更轻量可控。

#### 文档摄入参数

```yaml
ingestion:
  chunking:
    target-tokens: 240
    max-tokens: 320
    overlap-tokens: 32
  vector-batch-size: 9
document:
  ingestion:
    processing-timeout-minutes: 30
```


| 决策                               | 为什么                                                                       |
| -------------------------------- | ------------------------------------------------------------------------- |
| `target-tokens: 240`             | 约 60-80 个中文字，一个自然段大小。Alibaba text-embedding-v3 单次最多 512 tokens，240 留有足够余量 |
| `overlap-tokens: 32`             | 约 10 个中文字，保证切片边界处的语义连续性，避免关键信息落在切片间隙                                      |
| `vector-batch-size: 9`           | DashScope embedding 批量限制为 25，9 是保守值，留足重试和并发余量                             |
| `processing-timeout-minutes: 30` | 用于 StaleProcessingDocumentRecoveryRunner 判断"卡住的文档"阈值                      |


### 2.4 application-dev.yml 的关键差异


| 维度                  | local                   | dev                         |
| ------------------- | ----------------------- | --------------------------- |
| 端口                  | 10001                   | 10008                       |
| 数据库密码               | 明文                      | `${DB_PASSWORD}` 环境变量       |
| LLM API Key         | 内嵌 fallback             | `${DASHSCOPE_API_KEY}` 环境变量 |
| JWT secret          | 内嵌                      | `${JWT_SECRET}` 环境变量        |
| DevAdminInitializer | 不启动 (`@Profile("dev")`) | 启动                          |
| 日志格式                | ANSI 彩色                 | 纯文本                         |


**为什么有两套 profile**：local 用于本地开发（便利优先，内嵌凭据），dev 用于团队共享服务器（安全要求，凭据通过环境变量注入）。两套 profile 共享相同的结构和中间件地址，仅凭据注入方式不同。

### 2.5 Logback 配置

`logback-spring.xml` 在 application.yml 之前加载（Logback 在 Spring Boot 启动早期初始化）：


| Appender      | 目标                       | 格式         | 条件                                 |
| ------------- | ------------------------ | ---------- | ---------------------------------- |
| CONSOLE_COLOR | 控制台                      | ANSI 彩色    | `spring.profiles.active=local` 时启用 |
| CONSOLE_PLAIN | 控制台                      | 纯文本        | 非 local profile 时启用                |
| FILE          | `./logs/asklens.log`       | JSON Lines | 50MB/文件，保留 30 天，总上限 2GB            |
| FILE_ERROR    | `./logs/asklens-error.log` | 纯文本        | 仅记录 WARN+，保留 60 天，总上限 1GB          |


**为什么 Logback 用 `-spring` 后缀**：`logback-spring.xml` 而非 `logback.xml` — 前者由 Spring Boot 管理加载时机，支持 `<springProfile>` 标签根据 active profile 切换配置。

### 2.6 配置绑定验证

启动时 Spring Boot 对 `@ConfigurationProperties` 类执行 JSR-303 验证：

```java
@ConfigurationProperties("rag.auth")
@Validated
public class AuthProperties {
    @NotBlank private String issuer;
    @NotBlank private String jwtSecret;
    @Min(1) private int accessTokenExpireMinutes = 30;
    // ...
}
```

如果 `rag.auth.jwt-secret` 缺失或 `access-token-expire-minutes` 为 0，**启动立即失败并报错**，而非运行时 NPE。这是 Spring Boot 的"快速失败"哲学。

## 三、注解展开与 Bean 注册（阶段二：prepareContext）

### 3.1 @SpringBootApplication 的展开

`@SpringBootApplication` 等价于三个注解的组合：

```
@SpringBootApplication
    ├── @SpringBootConfiguration       → 标识这是一个配置类
    ├── @EnableAutoConfiguration       → 触发 Spring Boot 自动配置机制
    └── @ComponentScan("com.asklens") → 扫描当前包及子包下所有组件
```

#### @ComponentScan 扫描路径

以 `com.asklens` 为根，扫描所有子包下的 Spring 组件注解（`@Component`、`@Service`、`@Repository`、`@Controller`、`@RestController`、`@Configuration`）。共扫描到约 **46 个组件**，分布在以下模块：

```
com.asklens
├── auth/          # 3 @Component/@Service + 1 @Configuration
├── user/          # 2 @Service + 1 @Controller + 1 @RestController
├── common/        # 2 @RestControllerAdvice + 1 @Aspect + 1 @Configuration
├── document/      # 6 @Service + 1 @Controller + 1 ApplicationRunner
├── ingestion/     # 4 @Service + 1 @Configuration
├── engine/        # 4 @Service（检索引擎 + 对象存储）
├── group/         # 2 @Service + 1 @Controller
└── qa/            # 4 @Service + 1 @Controller + 2 @Configuration
```

### 3.3 @EnableAsync 的展开

```java
@EnableAsync
```

启用 Spring 的异步方法执行能力。Spring Boot 检测到该注解后自动配置 `AsyncTaskExecutor`（默认 `SimpleAsyncTaskExecutor`，Spring Boot 3.x 自动升级为 `TaskExecutionAutoConfiguration` 提供的 `ThreadPoolTaskExecutor`）。

**被 `@Async` 标记的方法**：`DocumentIngestionAsyncListener.handle()`。该 listener 监听文档上传完成事件，异步执行 ETL 流水线（解析 → 切片 → 向量化 → ES 索引），避免阻塞 HTTP 响应线程。

## 四、@Configuration 类加载与 Bean 实例化（阶段三：refreshContext）

### 4.1 加载顺序

Spring 不保证 `@Configuration` 类之间的实例化顺序，但 Bean **依赖关系**决定了实际初始化顺序：

```
1. AuthConfiguration            → 注册 Clock + PasswordHasher（无外部依赖，最先）
2. DocumentIngestionConfiguration → 注册 DocumentIngestionProcessor（依赖 ObjectProvider）
3. QueryPlanningConfiguration   → 注册 queryPlanningChatClient（依赖 ChatClient.Builder）
4. QaChatClientConfiguration    → 注册 qaChatClient + qaRetrievalAdvisor（依赖 ReadyChunkDocumentRetriever）
5. OpenApiConfiguration         → 注册 OpenAPI spec（无外部依赖）
```

### 4.2 AuthConfiguration

```java
@Configuration
@EnableConfigurationProperties(AuthProperties.class)
public class AuthConfiguration {

    @Bean
    public Clock authClock() {
        return Clock.systemDefaultZone();  // 可测试性：单元测试可注入 Clock.fixed()
    }

    @Bean
    public PasswordHasher defaultPasswordHasher() {
        return new BCryptPasswordHasher();  // BCrypt: 内置 salt，60 字符输出
    }
}
```

**为什么 Clock 是 Bean**：`System.currentTimeMillis()` 在代码中直接调用会导致测试不可控（无法模拟时间流逝）。将 `Clock` 作为 Bean 注入后，测试可以用 `Clock.fixed()` 冻结时间，验证 token 过期等时间敏感逻辑。

**为什么 BCrypt 而非 Argon2**：Spring Security Crypto 内置 BCrypt 实现，零额外依赖。BCrypt 虽非最先进，但经过二十余年验证，没有已知的安全漏洞。Argon2 需要额外的 JNI 绑定。

### 4.3 DocumentIngestionConfiguration

```java
@Configuration(proxyBeanMethods = false)
@EnableConfigurationProperties(ChunkingProperties.class)
public class DocumentIngestionConfiguration {

    @Bean
    public DocumentIngestionProcessor documentIngestionProcessor(
            ObjectProvider<DocumentParser> documentParsers,
            ObjectProvider<DocumentSplitter> documentSplitters,
            ObjectProvider<VectorIngestionService> vectorIngestionServices,
            ObjectProvider<TextCleanupTransformer> textCleanupTransformers
    ) {
        // ObjectProvider 延迟解析，避免循环依赖
        return new EtlDocumentIngestionProcessor(
            documentParsers.orderedStream().findFirst().orElseThrow(...),
            documentSplitters.orderedStream().findFirst().orElseThrow(...),
            vectorIngestionServices.orderedStream().findFirst().orElseThrow(...),
            textCleanupTransformers.orderedStream().findFirst().orElseThrow(...)
        );
    }

    @Bean
    @ConditionalOnMissingBean(TextCleanupTransformer.class)
    public TextCleanupTransformer textCleanupTransformer() {
        return new DefaultTextCleanupTransformer();
    }
}
```


| 设计点                           | 为什么                                                                                                                            |
| ----------------------------- | ------------------------------------------------------------------------------------------------------------------------------ |
| `proxyBeanMethods = false`    | 该配置中 Bean 之间不互相调用，关闭 CGLIB 代理提升启动速度                                                                                            |
| `ObjectProvider` 而非直接注入       | 各个 `DocumentParser`、`DocumentSplitter` 等 Bean 分散在 `ingestion` 包的不同类中，声明顺序不确定。`ObjectProvider` 延迟查找，避免 Spring 初始化顺序导致的循环依赖或缺失问题 |
| `orderedStream().findFirst()` | 选出优先级最高的实现。未来可以注册多个 Parser 处理不同格式（PDF vs DOCX），`@Order` 控制优先级                                                                  |
| `@ConditionalOnMissingBean`   | 允许其他模块（或未来扩展）注册自定义的 `TextCleanupTransformer` 覆盖默认行为，开闭原则                                                                       |


### 4.4 QaChatClientConfiguration

```java
@Configuration
public class QaChatClientConfiguration {

    @Bean
    public ChatClient qaChatClient(
            ChatClient.Builder chatClientBuilder,
            @Qualifier("qaSystemPromptTemplate") PromptTemplate qaSystemPromptTemplate,
            @Qualifier("qaRetrievalAdvisor") RetrievalAugmentationAdvisor qaRetrievalAdvisor
    ) {
        return chatClientBuilder
                .defaultSystem(qaSystemPromptTemplate.getTemplate())
                .defaultAdvisors(qaRetrievalAdvisor)
                .build();
    }

    @Bean
    public RetrievalAugmentationAdvisor qaRetrievalAdvisor(
            ReadyChunkDocumentRetriever documentRetriever
    ) {
        return RetrievalAugmentationAdvisor.builder()
                .documentRetriever(documentRetriever)
                .contextualQueryAugmenter(ContextualQueryAugmenter.builder()
                        .allowEmptyContext(true)
                        .build())
                .build();
    }

    @Bean
    public PromptTemplate qaSystemPromptTemplate() {
        return PromptTemplate.builder()
                .resource(new ClassPathResource("prompts/qa/system.st"))
                .build();
    }
    // ... qaUserPromptTemplate, qaRagContextPromptTemplate
}
```

**为什么 System Prompt 从文件加载而不是硬编码**：

1. Prompt 是经常迭代的内容，与 Java 代码解耦后修改 Prompt 无需重新编译。
2. 非技术人员（如产品经理）可以查看和调整 Prompt 模板。
3. 未来可以扩展为按群组/用户定制 System Prompt。

**为什么 QA ChatClient 独立于查询规划 ChatClient**：

- QA ChatClient 需要加载 System Prompt（角色约束 + JSON 输出格式）+ RAG Advisor（自动检索注入证据）。
- 查询规划 ChatClient 是纯净的 LLM 调用，不加任何 System Prompt 和 Advisor，只分析问题本身。
- 两者的配置需求完全不同，强行共用会导致职责混乱。

### 4.5 QueryPlanningConfiguration

```java
@Configuration
public class QueryPlanningConfiguration {

    @Bean("queryPlanningChatClient")
    public ChatClient queryPlanningChatClient(ChatClient.Builder chatClientBuilder) {
        return chatClientBuilder.build();  // 无 System Prompt，无 Advisor
    }

    @Bean("queryPlanningUserPromptTemplate")
    public PromptTemplate queryPlanningUserPromptTemplate() {
        return PromptTemplate.builder()
                .resource(new ClassPathResource("prompts/query-planning/user.st"))
                .build();
    }
}
```

**为什么用 `@Qualifier` 命名 Bean**：系统中存在两个 `ChatClient` Bean 和多个 `PromptTemplate` Bean，Spring 无法按类型区分。`@Qualifier` 明确标识每个 Bean 的用途，避免 DI 歧义性。

## 五、JwtAuthenticationFilter — 请求拦截链注册

### 5.1 注册机制

`JwtAuthenticationFilter` 标注了 `@Component`，继承 `OncePerRequestFilter`：

```java
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {
    // ...
}
```

Spring Boot 的 `WebMvcAutoConfiguration` 检测到容器中存在 `Filter` Bean，自动将其注册到 Servlet Filter Chain 中，无需显式调用 `addFilterBefore()`。

### 5.2 拦截逻辑

```
HTTP 请求进入
    ↓
shouldNotFilter(request)?
    ├── 是（auth/login, /register, /refresh, /logout）→ 放行
    └── 否
        ↓
    从 Authorization header 提取 Bearer token
        ├── 无 token → 401 {"success": false, "message": "未提供认证凭证"}
        ├── token 无效/过期 → 401 {"success": false, "message": "认证凭证无效或已过期"}
        ├── 用户被禁用 → 403 {"success": false, "message": "账号已被禁用"}
        └── 验证通过 → UserContext.set(user) → chain.doFilter()
```

### 5.3 UserContext 的 ThreadLocal 设计

```java
public class UserContext {
    private static final ThreadLocal<AuthenticatedUser> currentUser = new ThreadLocal<>();

    public static void set(AuthenticatedUser user) { currentUser.set(user); }
    public static AuthenticatedUser get() { return currentUser.get(); }
    public static void clear() { currentUser.remove(); }
}
```

**为什么用 ThreadLocal**：

- Servlet 容器（Tomcat）为每个请求分配一个线程，ThreadLocal 天然地将用户身份绑定到当前请求线程。
- 无需在 Controller 方法签名中传递 `HttpServletRequest`（CurrentUserService 通过 `UserContext.get()` 即可获取）。
- Filter 的 `finally` 块中执行 `clear()`，防止线程归还线程池后残留用户信息（内存泄漏 + 安全隐患）。

**为什么不直接用 Spring Security 的 SecurityContextHolder**：

- 项目规模不需要 Spring Security 全套（URL 级鉴权、角色体系、Session 管理），引入后反而增加配置复杂度。
- 一个 150 行的 `OncePerRequestFilter` + ThreadLocal 足以覆盖当前需求（JWT 解析 → 设置当前用户）。
- 未来如果需要更复杂的权限模型，可以平滑迁移到 Spring Security（ThreadLocal 改成 SecurityContextHolder 的 strategy）。

## 六、ApplicationRunner 执行（阶段四：callRunners）

`ApplicationRunner.run()` 在 `refreshContext()` 完成后、`ApplicationReadyEvent` 发布前执行。多个 Runner 按 `@Order` 排序。

### 6.1 StaleProcessingDocumentRecoveryRunner（常驻，所有 profile）

```java
@Component
public class StaleProcessingDocumentRecoveryRunner implements ApplicationRunner {

    private final DocumentMapper documentMapper;
    private final int processingTimeoutMinutes;

    public StaleProcessingDocumentRecoveryRunner(
            DocumentMapper documentMapper,
            @Value("${document.ingestion.processing-timeout-minutes:30}") int processingTimeoutMinutes
    ) {
        this.documentMapper = documentMapper;
        this.processingTimeoutMinutes = processingTimeoutMinutes;
    }

    @Override
    @Transactional
    public void run(ApplicationArguments args) {
        LocalDateTime timeout = LocalDateTime.now().minusMinutes(processingTimeoutMinutes);
        int count = documentMapper.markStaleProcessingAsFailed(timeout);
        if (count > 0) {
            log.warn("Recovered {} stale documents, marked as FAILED", count);
        }
    }
}
```

XML 中的 SQL：

```sql
UPDATE documents
SET status = 'FAILED', failure_reason = '服务器异常重启，处理中断',
    processed_at = NOW(), updated_at = NOW()
WHERE status = 'PROCESSING'
  AND (processed_at IS NULL OR processed_at < #{timeout})
```

**为什么需要这个 Runner**：

文档上传完成后的 ETL 流程是异步的（`@Async`），文档状态机为：

```
UPLOADING → PROCESSING → READY
                      ↘ FAILED（重试耗尽）
```

如果服务器在文档处于 `PROCESSING` 状态时异常关机（kill -9、OOM、断电），该文档将**永久卡在 PROCESSING 状态**。重启后这些文档既不会被自动重试，用户也无法重新上传（同名文件被 MinIO 占用），也无法手动重试（没有提供相应的 API）。

StaleProcessingDocumentRecoveryRunner 在**每次启动**时扫描并修复这种"僵尸状态"的文档：

1. 查询所有 `status = 'PROCESSING'` 且 `processed_at` 早于 30 分钟前（或为 NULL）的文档
2. 批量更新为 `FAILED`，`failure_reason` 写明原因
3. 用户可以在前端看到失败状态，调用 `/api/documents/{id}/retry-ingestion` 重新触发

**为什么超时阈值是 30 分钟**：

- 正常文档 ETL 流程（解析 PDF + 切片 + 向量化 + ES 索引）在 10MB PDF 下约需 30 秒~2 分钟。
- 30 分钟是 15 倍安全余量，覆盖极端情况（超大文件、Embedding API 限流重试）。
- 可配置：`document.ingestion.processing-timeout-minutes`，不同环境可设不同值。

### 6.2 DevAdminInitializer（仅 dev profile）

```java
@Component
@Profile("dev")  // 仅在 dev profile 生效
public class DevAdminInitializer implements ApplicationRunner {

    @Override
    public void run(ApplicationArguments args) {
        // 检查 admin 用户是否存在
        // 不存在 → 创建（密码 BCrypt 加密）
        // 存在 → 更新密码为最新配置值（确保开发环境始终可用 admin 登录）
    }
}
```

**为什么只在 dev 生效**：

- local profile 由开发者在本地手动创建 admin 账户。
- dev 是共享环境，多人使用。每次启动强制刷新 admin 密码，避免"有人改了密码导致其他人登不上"的情况。
- 生产环境不应存在此逻辑（安全风险）。

**为什么用 ApplicationRunner 而非 @PostConstruct**：  
`@PostConstruct` 在 Bean 初始化阶段执行，此时 DataSource 可能尚未完全就绪（连接池正在填充）。`ApplicationRunner` 保证在 DataSource、MyBatis-Plus、TransactionManager 全部就绪后执行，SQL 操作安全。

## 七、应用就绪

### 7.1 ApplicationReadyEvent

Spring Boot 在 `callRunners()` 完成后发布 `ApplicationReadyEvent`：

```
SpringApplication.run() 内部
    → prepareEnvironment()
    → prepareContext()
    → refreshContext()
    → afterRefresh()         ← 此时 ApplicationRunner.run() 已执行完毕
    → listeners.started()
    → callRunners()
    → listeners.ready()      ← 发布 ApplicationReadyEvent
```

当前代码中没有 `@EventListener(ApplicationReadyEvent.class)` 的监听器。但事件机制存在，未来可以扩展（如启动后预热向量索引、预加载 Prompt 模板缓存等）。

### 7.2 @Async 事件监听就绪

`DocumentIngestionAsyncListener` 虽然不是一个 Runner，但它是 `@Component` + `@TransactionalEventListener`，在容器启动后即可接收事件：

```java
@Component
public class DocumentIngestionAsyncListener {

    @Async
    @TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
    public void handle(DocumentIngestionRequestedEvent event) {
        documentIngestionAsyncService.ingestDocument(event.documentId(), event.groupId());
    }
}
```

`phase = AFTER_COMMIT` 保证事件处理在 `DocumentUploadService`（或 `DocumentDeleteService`）的事务提交成功后触发——如果数据库写入失败回滚，不会触发无效的异步 ETL。
