## 1. 现状摘要

| 模块 | 现状 | 主要缺口 |
|------|------|----------|
| 文档中心 | V3 已完成上传 / ETL / 预览 / 重试 | 尚未支持分片上传 UI（后端接口齐全）、缺少归档与标签 |
| 协作小组 | V2 的邀请、加入、管理齐备 | 会员权限仍是二元（OWNER / MEMBER），无细粒度 |
| 知识库问答（QA） | `POST /api/qa/ask` 单次一问一答 | **无会话 / 无历史 / 前端内存态**，刷新即丢 |
| AI 助手（Assistant） | V4 已接入 Agent、流式、会话、短期记忆 | 与 QA 模块重复度高（两套问答路径） |
| 账户与鉴权 | JWT + Refresh Token Cookie | 无 2FA、无第三方登录 |

## 2. QA 会话持久化（重点）

### 2.1 问题定义

当前 QA 页面（[Argus-frontend/src/views/qa/](../Argus-frontend/src/views/qa/)）实现了 ChatGPT 风格的多会话交互，但**会话仅保存在前端 `ref` 中**（参见 [useQaSessions.ts](../Argus-frontend/src/views/qa/composables/useQaSessions.ts)）。刷新页面、换浏览器、换设备均会丢失。

背景成因：后端 [QaController](../Argus-backend/src/main/java/com/argus/rag/qa/controller/QaController.java) 仅暴露 `POST /api/qa/ask` 单次问答接口，没有 `session`、`history`、`citation` 的持久化通道，前端只能作为"一次性请求"使用。

V4.0 的 Assistant 模块已经实现了会话持久化 / 短期记忆 / 流式事件的完整骨架（[assistant_sessions](../sql/schema.sql) / [assistant_messages](../sql/schema.sql)），QA 模块的持久化应**复用同一套设计思路**，但保持**路由隔离**（避免两套会话混淆）。

### 2.2 设计目标

- 用户的 QA 对话**跨设备、跨会话、跨刷新**不丢失
- 证据引用（`citations`）与答案一同持久化，回看历史时仍能看到每个 chunk 的来源
- 会话按所属群组隔离，权限模型与 QA 查询本身一致（群组成员可读、仅本人可写/删）
- 提供轻量的"列表 / 详情 / 删除 / 归档"接口，与 Assistant 模块的 API 保持形态一致，便于前端复用组件

### 2.3 数据模型

新增两张表：`qa_sessions`、`qa_messages`。命名与结构对齐 `assistant_sessions` / `assistant_messages`，便于开发者快速理解。

```sql
-- ============================================================
-- QA 会话表（按用户 + 群组维度划分对话上下文）
-- ============================================================
CREATE TABLE IF NOT EXISTS qa_sessions (
    id               BIGSERIAL     PRIMARY KEY,
    user_id          BIGINT        NOT NULL,
    group_id         BIGINT        NOT NULL,
    title            VARCHAR(255)  NOT NULL DEFAULT '新的问答',
    status           VARCHAR(32)   NOT NULL DEFAULT 'ACTIVE',
    last_message_at  TIMESTAMP,
    message_count    INT           NOT NULL DEFAULT 0,
    created_at       TIMESTAMP     NOT NULL DEFAULT now(),
    updated_at       TIMESTAMP     NOT NULL DEFAULT now(),

    CONSTRAINT fk_qa_session_user  FOREIGN KEY (user_id)  REFERENCES users (id),
    CONSTRAINT fk_qa_session_group FOREIGN KEY (group_id) REFERENCES groups (id)
);

COMMENT ON TABLE  qa_sessions                IS '知识库问答会话';
COMMENT ON COLUMN qa_sessions.status         IS 'ACTIVE | ARCHIVED | DELETED';
COMMENT ON COLUMN qa_sessions.message_count  IS '冗余列，避免列表页 N+1 count';

-- 列表查询主索引（用户维度 + 状态 + 时间倒序）
CREATE INDEX IF NOT EXISTS idx_qa_sessions_user_status_time
    ON qa_sessions (user_id, status, last_message_at DESC);

-- ============================================================
-- QA 消息表（角色化存储，citations 走 JSONB）
-- ============================================================
CREATE TABLE IF NOT EXISTS qa_messages (
    id            BIGSERIAL     PRIMARY KEY,
    session_id    BIGINT        NOT NULL,
    role          VARCHAR(16)   NOT NULL,          -- USER | ASSISTANT
    content       TEXT          NOT NULL,
    answered      BOOLEAN,                          -- 仅 ASSISTANT 有效
    reason_code   VARCHAR(64),                      -- 拒答码，如 NO_EVIDENCE / LOW_CONFIDENCE
    reason_message TEXT,
    citations     JSONB,                            -- 引用数组
    latency_ms    INT,                              -- 端到端耗时，便于可观测
    created_at    TIMESTAMP     NOT NULL DEFAULT now(),

    CONSTRAINT fk_qa_message_session FOREIGN KEY (session_id) REFERENCES qa_sessions (id)
);

COMMENT ON COLUMN qa_messages.citations IS 'CitationItem 数组，字段与 /api/qa/ask 响应一致';

-- 按会话 ID + 时间正序（拉取消息流）
CREATE INDEX IF NOT EXISTS idx_qa_messages_session_time
    ON qa_messages (session_id, created_at);
```

设计要点：

- **一个用户 + 一个群组**允许多个 `qa_session`，与当前前端"新建问答"的行为对齐（用户可以针对同一个知识库开多个独立话题）。
- `citations` 以 `JSONB` 存储。原因：`CitationItem` 是只读快照，没有跨行查询需求；JSONB 可索引但暂不需要。
- `message_count` 冗余列在保存消息时同步 `+1`，避免列表页为每个会话执行一次 `COUNT(*)`。
- 删除走**软删除**（`status = DELETED`），保留恢复窗口（与 Assistant 模块一致）。

### 2.4 接口设计

所有接口走 `/api/qa/sessions` 前缀，与现有 `/api/qa/ask` 并列；保持与 `/api/assistant/sessions` 形态一致，前端组件层可以直接复用。

| Method | 路径 | 语义 | 权限 |
|--------|------|------|------|
| `GET` | `/api/qa/sessions` | 列出当前用户会话，可按 `groupId`、`status` 过滤 | 登录用户 |
| `POST` | `/api/qa/sessions` | 创建空会话（指定 `groupId`） | 群组成员 |
| `GET` | `/api/qa/sessions/{sessionId}` | 会话详情 + 消息列表（分页） | 会话所有者 |
| `PATCH` | `/api/qa/sessions/{sessionId}` | 重命名 / 归档 | 会话所有者 |
| `DELETE` | `/api/qa/sessions/{sessionId}` | 软删除 | 会话所有者 |
| `POST` | `/api/qa/sessions/{sessionId}/ask` | **在会话内提问**（替代旧的 `/api/qa/ask`） | 会话所有者 |

关键改造点：**提问入口从 `/api/qa/ask` 迁移到 `/api/qa/sessions/{id}/ask`**，其余参数（`question`）保持不变。旧接口保留 2 个版本做兼容过渡，打 `@Deprecated` 标记。

请求 / 响应示例（以"在会话内提问"为主）：

```http
POST /api/qa/sessions/42/ask
{
  "question": "文档分片的 chunk size 如何配置？"
}
```

```json
{
  "sessionId": 42,
  "userMessageId": 1001,
  "assistantMessageId": 1002,
  "answered": true,
  "answer": "在 application.yml 中通过 argus.ingestion.chunk-size ...",
  "reasonCode": null,
  "reasonMessage": null,
  "citations": [ { "documentId": 7, "chunkId": 88, "fileName": "config-guide.md", "score": 0.92, "snippet": "..." } ],
  "latencyMs": 1830
}
```

对比当前 `AskQuestionResponse`，新增 4 个字段：`sessionId` / `userMessageId` / `assistantMessageId` / `latencyMs`。前者给前端做乐观更新，后者是可观测性埋点。

### 2.5 后端分层

沿用项目现有分层约定（controller / service / mapper / entity / dto / vo）：

```
com.argus.rag.qa
├── controller
│   ├── QaController                 // 旧接口（保留过渡）
│   └── QaSessionController          // 新增：会话 CRUD + ask
├── service
│   ├── QaService                    // 旧 ask 编排（保留）
│   ├── QaSessionService             // 新增：session 生命周期
│   ├── QaConversationService        // 新增：消息读写（含自动命名）
│   └── QaChatService                // 已有：检索 + 生成
├── mapper
│   ├── QaSessionMapper
│   └── QaMessageMapper
└── model
    ├── entity
    │   ├── QaSessionEntity
    │   └── QaMessageEntity
    ├── dto                          // Ask / CreateSession / RenameSession
    └── vo                           // QaSessionItemVO / QaMessageVO / AskInSessionResponse
```

权限校验收口到 `QaSessionService`：
- `requireSessionOwner(userId, sessionId)` — 会话级 CRUD 入口
- `requireGroupReadable(userId, groupId)` — 创建会话时校验群组成员身份（复用 `GroupMembershipService`）

### 2.6 前端改造

改动主要集中在 [src/views/qa/](../Argus-frontend/src/views/qa/)：

1. **新增 API 层** [src/api/qa.ts](../Argus-frontend/src/api/qa.ts)：`listQaSessions / createQaSession / getQaSession / renameQaSession / deleteQaSession / askInSession`。
2. **重写 composable** [useQaSessions.ts](../Argus-frontend/src/views/qa/composables/useQaSessions.ts)：
   - 把 `sessions: Ref<QaSession[]>` 的数据源从内存切换为远端
   - 初次加载调 `listQaSessions`；切换会话时调 `getQaSession` 拉消息
   - `askInSession` 返回后把 `{ userMessageId, assistantMessageId }` 回填到本地数组
3. **侧边栏** [QaSidebar.vue](../Argus-frontend/src/views/qa/components/QaSidebar.vue)：
   - 文案从"会话仅保存在当前浏览器"改为"已同步至云端"
   - 增加"归档"入口（三点菜单）
4. **URL 同步**：`/app/qa?session=42` — 让会话可分享 / 可书签。

### 2.7 观测与安全

- 新增 Prometheus 指标：`qa_ask_latency_ms`（按 `answered` 打标签）、`qa_session_total{user_id}`。
- 日志结构化字段：`sessionId`、`userMessageId`、`answered`、`reasonCode`，便于追踪"为什么用户觉得某次回答不准"。
- 软删除的会话设置 **保留 30 天**，之后由定时任务 `QaSessionPurgeJob` 物理清理。

### 2.8 工作量估算

| 阶段 | 产出 | 估时 |
|------|------|------|
| 表结构 + Flyway 迁移 | `sql/schema.sql` + `V__add_qa_sessions.sql` | 0.5 人日 |
| 后端 CRUD + 会话内 ask 编排 | `QaSessionController` / Service / Mapper / DTO / VO | 2 人日 |
| 后端单元 + 集成测试 | `@SpringBootTest` 覆盖权限、自动命名、citations 持久化 | 1 人日 |
| 前端改造 | API + composable + 侧边栏 + URL 同步 | 1.5 人日 |
| 联调 + 旧接口废弃公告 | `@Deprecated` + 文档更新 | 0.5 人日 |
| **合计** | | **≈ 5.5 人日** |

### 2.9 风险与兜底

- **风险 1：旧接口使用方**。旧 `/api/qa/ask` 可能被外部脚本调用。对策：保留旧接口至少 2 个大版本，日志里统计调用来源。
- **风险 2：`citations` 字段膨胀**。对策：JSONB 存储前做字段白名单裁剪（`snippet` 截断到 500 字符），保留展示必要内容。
- **风险 3：与 Assistant 模块的语义混淆**。对策：保持路由前缀清晰（`/api/qa/...` vs `/api/assistant/...`），表也分开（`qa_*` vs `assistant_*`），短期不合并。

## 3. 其他改造项一览

按优先级排序，每项只列"问题 - 方案 - 估时"核心三要素。

### 3.1 QA 与 Assistant 模块的长期收敛（高优）

**问题**：V3 的 QA 与 V4 的 Assistant 均实现了"基于群组知识库回答"。两者有两套会话表、两套检索路径，长期会造成语义断裂（同一问题在两个入口里答案可能不同）。

**方案**：QA 会话持久化（本文档 §2）完成后，评估把 `QaChatService` 收拢进 Assistant 的 `KB_SEARCH` 工具模式，让 QA 页面从底层调用 `AssistantService`，仅在 UI 层保留"纯知识库提问"的精简形态。

**估时**：调研 1 人日 + 收敛改造 3 人日。**前置依赖：§2 已落地**。

### 3.2 文档分片上传的前端落地（中优）

**问题**：后端 `/api/documents/upload/init | chunks | complete` 已完整实现，当前前端只用单次 `/upload`（10MB 上限）。

**方案**：前端新增 `useChunkedUpload` composable，按 4MB 切片 + 秒传哈希判断 + 断点续传。

**估时**：2 人日。

### 3.3 群组成员细粒度权限（中优）

**问题**：当前仅 `OWNER` / `MEMBER` 二元权限，无法表达"只读旁观者"或"可以上传但不能删除"。

**方案**：引入 `GroupRole` 枚举扩展（`OWNER / EDITOR / VIEWER`）+ 基于注解的 `@RequireGroupRole` 方法拦截器。

**估时**：3 人日（含迁移脚本 + 旧数据默认映射为 EDITOR）。

### 3.4 文档标签与归档（低优）

**问题**：文档只有文件名，无主题分类。大群组查找困难。

**方案**：新增 `document_tags` 多对多关联，文档中心筛选器增加标签维度。

**估时**：2 人日。

### 3.5 登录安全增强（低优 / 独立）

**问题**：无 2FA，无登录异常告警。

**方案**：接入 TOTP（Google Authenticator）+ 登录 IP 风控表。

**估时**：3 人日（非阻塞迭代，可排在任意时点）。

### 3.6 可观测性基建（贯穿性）

**问题**：目前只有应用日志，缺少结构化指标采集、分布式 trace。

**方案**：集成 Micrometer + Prometheus + OpenTelemetry SDK；先保证 QA、Ask、文档 ETL 三条关键链路打点。

**估时**：2 人日（建议与 §2 同步做，QA 会话的指标直接并入）。
